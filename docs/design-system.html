design system · handoff

The Tavern design system.

A complete reference for the visual, motion, sound, and copy language of Tavern. Source of truth for every UI decision. Read this before designing or editing anything in apps/web.

00

For Claude Code

This document is the canonical reference for any agent doing UI work on Tavern. Read in this order, then check your work against the do/don'ts in each component's section.

Read order

  1. CLAUDE.md — terse project conventions. Always loaded into your context first.
  2. This document — the visual + interaction system. Skim once; jump back to relevant sections per task.
  3. The component / route sections — when you need to see how a specific surface is shaped. The map lists every section.

Workflow

  1. Map the request to a surface. Most things already have a phase doc. Use the surface index.
  2. Use existing components. Reach for Modal + atoms, MessageList, etc. Don't build new ones unless the inventory genuinely doesn't cover the case.
  3. Use semantic tokens only. bg-canvas, fg-default, accent-ember. Never the deprecated tavern-* classes.
  4. Use the named transitions. --t-base + --ease-decel for enter; --t-base + --ease-accel for exit. See Motion.
  5. Stay in voice. See Voice & copy before writing user-facing strings.
  6. Check do/don'ts. Every component section has them.

What this system does not cover

  • Backend / API shape. NotificationInbox, InviteModal, and SettingsModal have placeholder endpoints — wire them when the API ships.
  • OAuth flows. SSO buttons are inert. Wire to auth.ts.
  • Mobile-native concerns. The web app is responsive but not a PWA.
  • Internationalization. All copy is English; no i18n scaffolding yet.
!
If you find yourself inventing a colour, font, or transition — stop. Either it already exists as a token, or it's wrong. Search this doc, then ask.
01

Principles

Four ideas that shape every decision below. Use them as a tiebreaker.

  1. Warmth, not friction. Tavern is a place to gather. Surfaces are warm-dark, not pure black. Type is human, not industrial. Motion confirms intent rather than performing.
  2. Restraint as identity. One serif, one sans, one mono. One primary accent. Five sounds. Five durations. Constraints are the brand.
  3. Earned attention. Pings, badges, sound, and motion are scarce resources. The rarer they are, the more they mean.
  4. Hospitable copy. Every label, every error, every empty state is written like a host speaking to a guest — not a system speaking at a user.

Tints --tint-*

14% mix of an accent into transparent. Use for hover backgrounds, active row fills, badge surfaces — anywhere an accent needs to live as a background without overwhelming. Use bg-surface on top of bg-canvas to express that a card sits above the page.

Italic serif

Reserved for the brand voice in display headlines: "the warmth in between". Don't use italic body for emphasis in UI copy — use weight instead.

Use serif for every member's display name in messages and member lists. It's part of the warmth.

don't

Use serif for buttons, inputs, or short-burst UI text. Sans handles all UI.

Voice & copy

Warm, calm, a little literary. Write as a host speaking to a guest. Use the metaphor sparingly — "tavern", "room", "table", "chair", "doors" — never "channel" / "server" / "join" in user-facing copy.

Lexicon

UseNot
TavernServer, workspace, guild
RoomChannel (only in code)
Member · guest · regularUser
Pull up a chair · Open the doorsSign up · Get started
Around the tableActive voice channel
A chair is waitingWelcome to your account
Light the hearthCreate your first instance

Tone shifts

  • Empty states — gentle and inviting. "All quiet for now." not "No notifications."
  • Errors — calm and specific. Tell them what happened and what to do. Never blame the user. Never use "Oops" or "Whoops".
  • Success — quick and warm. "Saved." not "Settings updated successfully."
  • Destructive confirmations — direct, not dramatic. "Remove from the tavern?" not "ARE YOU SURE you want to PERMANENTLY DELETE?"
do

"We couldn't reach the tavern. Check your connection, or try again in a moment."

don't

"Error: Network request failed (code 503). Please retry."

Mechanics

  • Sentence case for headings, labels, buttons. Never title case.
  • One period per sentence, including button labels with two clauses. ("Save changes." not "Save changes")
  • Oxford comma. Always.
  • Numbers: spell out under 10 in body copy; numerals in UI labels and counts.
  • Time: "Tue 8:14p" (mono, terse). "4m ago" / "3 weeks ago" for relative.

Hard rules. fg-faint is for kickers, timestamps, and decorative icons only — never for paragraphs, never for interactive labels. Use it only on bg-canvas, bg-sunken, or bg-surface, and only at sizes ≥16px regular / ≥14px bold. Use fg-muted the moment a string is part of a sentence.

Focus management

  • Visible focus ring — border-strong 2px outline + 2px offset. Never outline:none.
  • Modals + popovers + drawers trap focus when open and restore it on close. Radix Dialog handles this for Modal and ReportDialog; the inbox popover does it via the useFocusTrap helper.
  • Esc closes the topmost overlay (popover before drawer before modal). Never closes more than one layer per press.
  • Skip link at the top of AppShell jumps to the message column, so keyboard users don't tab through the rail every navigation.

ARIA patterns

  • MessageList — role="log" with aria-live="polite" and aria-atomic="false". New incoming messages are announced once; the toolbar that appears on hover is decorative and gets aria-hidden.
  • Mention arrival — fires an aria-live="assertive" announcement: "{author} mentioned you in #{room}."
  • Modal — Radix Dialog provides the right role + labelledby/describedby plumbing. Always pass title and description to Modal so it's wired correctly.
  • NotificationInbox — the bell button is aria-haspopup="dialog" and reflects open state via aria-expanded. Tabs use role="tablist" + role="tab" + role="tabpanel".
  • VoiceRoom — participant tile chips are role="status" so changes (joined / muted / speaking) are announced quietly. Speaker outline is decorative.
  • TypingIndicator — wrapped in a node with role="status" and aria-live="polite"; pulsing dot is aria-hidden.

Reduced motion

Already specified in Motion — when prefers-reduced-motion is set, spring/emphasized collapse to standard, cinematic compresses to base, and pulse / clatter / swipe drop their motion entirely. Components

Each component lives in apps/web/src/components/ (or apps/web/src/routes/ for shell/auth) and is the canonical implementation. Reach for these before making new components.

AppShell

routes/app-shell.tsx

The root product layout. Hosts the server rail, channel sidebar, message outlet, and member sidebar. Owns the responsive drawer mechanics for mobile and tablet.

Anatomy

  • Server rail — 72px fixed left column. Renders ServerRail.
  • Channel sidebar — 240px column right of the rail. Renders ChannelSidebar.
  • Outlet — flex-fills with the active route (message list, settings, etc.).
  • Member sidebar — 240px right column. Renders MemberSidebar. Hideable.

Key props

type AppShellProps = {
  activeServerId: string | null;
  activeChannelId: string | null;
  // the outlet renders the active route
};

Routing contract

Reads useParams() for serverId and channelId. Reads useAuth() for current user. Reads useRealtime() for presence + ws plumbing. Don't add new context dependencies — extend the existing hooks.

do

Mount new top-level overlays (modals, lightbox) as siblings inside AppShell, gated by local state.

don't

Mount new persistent UI inside the outlet — it should appear in the shell, not move with route changes.

ServerRail

inline in routes/app-shell.tsx

The 72px vertical strip of server icons. Active server uses the morph transition (circle → 14px radius) plus a 3×28px ember edge bar to its left.

  • Edge bar — absolutely positioned, -left-3 top-1/2 h-7 w-[3px], ember. Grows in via the morph transition.
  • Add-server button — same dimensions, dashed border, fg-faint plus glyph.

MessageList & Message

components/MessageList.tsx

The scrolling list of messages and a single message row. Renders day dividers, author blocks (grouped consecutive messages from the same author), inline reactions, and a hover-only action toolbar.

Author block rhythm

  • First message in a block: avatar + serif display name (in role colour) + mono timestamp + body.
  • Subsequent messages: no avatar, no name, no timestamp; body only with 5px top padding. Timestamp appears in the gutter on hover.
  • New block starts when: author changes, or ≥ 7 minutes elapse, or a day divider falls between.

Inline elements

  • Mention chip — ember tint background, ember text, mono @handle. Hover lifts a member hovercard (not yet built).
  • Channel reference — mead tint, mead text, mono #room.
  • Reaction — pill with emoji + count + mono. Active (you reacted) uses tint-ember + ember border.
  • Edited marker — mono (edited) in fg-faint after the body.

Hover action toolbar

A floating bar of icon buttons appears top-right of the hovered message: react · reply · thread · more. Uses the lift transition. Pinned messages get a mead corner glyph instead of/in addition to the toolbar.

Day divider

/* mono kicker centred on a hairline rule */
TUESDAY · FEB 12
do

Use the role colour from the member's primary role for the display name. Owner=ember, Mod=mead, DM=lavender, regular=fg-default, etc.

don't

Colour the message body. Only the display name takes role colour. Bodies are always fg-default.

ReactionBar

components/ReactionBar.tsx

The row of reaction pills that hangs under a message, plus the quick-emoji popover for adding new ones. Renders only when the message has reactions or is hovered.

Anatomy

Modal & atoms

components/Modal.tsx

The shared modal shell plus a set of form atoms used inside every modal and auth route. Use these — don't reinvent.

Modal props

type ModalProps = {
  open: boolean;
  onOpenChange: (open: boolean) => void;
  title: string;
  kicker?: string;          // e.g. "I · Tavern"
  description?: string;
  size?: 'sm' | 'md' | 'lg';  // 420 / 540 / 720
  footer?: ReactNode;
  children: ReactNode;
};

Exported atoms

AtomWhat
FieldLabelled wrapper. label, hint, error.
TextInputStandard input. Auto-focus when first field of an opening modal.
TextAreaMultiline. Auto-grows 2–6 rows.
PrimaryButtonEmber background, on-accent text. Disabled at 40% opacity.
GhostButtonTransparent + border-default. For Cancel / secondary.
ErrorAlertRust-tinted panel. title + message.

Structural rules

  • Header carries kicker → title → description → close, in that order, with a border-bottom rule.
  • Body scrolls at max-height: 60vh. Internal padding var(--space-5) (20px).
  • Footer always lives on bg-sunken with a border-top rule. Actions right-aligned. Primary action is rightmost.
  • Backdrop is bg-black/70 with backdrop-blur-sm. Click closes unless preventOutsideClose.
do

Compose every form modal with <Field><TextInput …></Field>. Field handles label, hint, and error spacing consistently.

don't

Build a modal with a custom backdrop or a different footer layout. Even the auth screens — which aren't modals — match this rhythm.

Create… modals

components/Create{Server,Channel,Campaign}Modal.tsx

Three concrete modals built on the shared Modal shell + form atoms. They are the canonical example of how to compose a form modal in this app — copy their shape when adding a new one.

Common shape

Onboarding & auth

routes/register.tsx · routes/login.tsx · routes/bootstrap-page.tsx

Three routes, one visual language. Centred column on warm canvas, serif headline, mono kicker, ember primary CTA. All errors use the rust ErrorAlert pattern.

Register flow

Four-step stepped form with a top stepper. Each step has a bottom-anchored nav row (Back · Primary). Inline validation per step.

  1. Invite — SSO buttons (placeholder), divider, invite code (mono, uppercase).
  2. Account — email + password with show/hide eye, four-segment strength meter (rust → mead → moss).
  3. Identity — @username with sanitisation, display name, live preview card showing how the avatar + name will appear in chat.
  4. Welcome — ember check, serif "Welcome, {name}.", CTA "Enter the hall" → routes to /app.

Login

Single screen. SSO row above the password form. Show/hide password. Inline "Forgot?" link beside the password label.

Bootstrap (first-run)

Single screen, only reachable when /auth/bootstrap-status returns needsBootstrap: true. Two grouped cards: The keeper (display name, username, email, password) and The room (first server name). CTA: "Open the doors".

!
Bootstrap is one-time. Once any user exists, the endpoint refuses further calls and the route redirects to login. Don't add post-bootstrap state to this route — put it in an onboarding step instead.

i
Permission gating. Each route checks the relevant permission bitflag at mount and redirects to the channel page with a rust toast ("You don't have access to that.") on denial. The check happens in the route component, not the shell — keep it that way so the shell stays presentational.
